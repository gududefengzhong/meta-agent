# STATUS_ANNOTATION.md - 状态标注规范

## 文档定位
本文件定义本仓库所有文档、目录结构、命令入口、部署清单的「当前/目标/候选」三态标注约定。

凡涉及目录结构、模块边界、命令入口、部署形态、对外接口的描述，**必须明确标注其状态**，禁止把目标态描述成既成事实。

本文件优先级低于 `CLAUDE.md` 与 `AGENT_SPEC.md`，用于补充其「当前状态标注要求」条款。

## 三态定义

### 【当前】（implemented）
- 已经在主分支中落地、可运行、可验证的实现。
- 行为与文档描述一致；具备自动化测试或可手工复跑步骤。
- 示例：`【当前】仓库仅包含 docs/specs/ 下的规范文档，无任何 Python 源码。`

### 【目标】（target）
- 规划中要达到的形态，尚未在仓库中完整实现。
- 表达「将要走的方向」，不可作为现状引用。
- 示例：`【目标】生产部署采用 K8s + Helm Chart，API 与 Worker 均多副本。`

### 【候选】（candidate）
- 多个备选方案尚未拍板时的中间状态。
- 必须列出候选项与权衡维度，避免文档中出现「暗自决定」的事实。
- 示例：`【候选】任务队列在 Redis Streams / NATS JetStream / Kafka 之间未决，详见 INFRA_SELECTION_MATRIX。`

## 强制标注的场景
- 目录结构与模块边界
- 命令入口、CLI 子命令、API 路径
- 部署拓扑、容器组成、依赖中间件
- 对外接口（REST / MCP / webhook / 其他集成边界）
- 配置项、环境变量、Secret 字段
- 性能 / 容量 / 可用性数字
- 选型方案（数据库、队列、缓存、限流、熔断、Trace 后端等）

## 不需要标注的场景
- 引用外部规范、行业事实陈述
- 开发原则、协作规则、约束条款本身（这些不是实现态）

## 标注书写规则
- 标注前缀紧贴段落首行：`【当前】...` / `【目标】...` / `【候选】...`。
- 同一段同时含已实现与目标行为时，必须拆分为独立段落。
- 表格中使用单独列，或在单元格首部加标注前缀。
- PR 与提交说明里涉及目录 / 接口 / 部署描述，必须沿用本约定。

## 状态变更流程
- 【候选】→【目标】：在 PR 中说明决策与依据，并在 `AGENT_SPEC.md` 相应段落收口。
- 【目标】→【当前】：PR 必须附带可验证手段（自动化测试、可运行示例、文档复跑步骤）。
- 【当前】→【目标/候选】：属于回退或重新设计，必须在文档中显式记录原因与影响范围。

## 反面示例
- ❌ 「项目采用 `core/`、`adapters/`、`infra/` 目录结构。」（实际尚未落地）
- ✅ 「【目标】项目采用 `core/`、`adapters/`、`infra/` 目录结构；【当前】仅 `docs/` 与 `CLAUDE.md` 存在。」

- ❌ 「系统支持 1000 QPS、P99 200ms。」（未经容量验证）
- ✅ 「【目标】具体 QPS / 延迟指标在实施阶段结合容量规划单独固化。」
